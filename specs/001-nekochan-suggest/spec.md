# 機能仕様書: ネコチャン絵文字提案ツール

**Feature Branch**: `001-nekochan-suggest`  
**Created**: 2026-03-19  
**Status**: Draft  
**Input**: User description: "文章に対して、その文章に追加するおすすめのネコチャン絵文字を提案するツールを作る"

## ユーザーシナリオとテスト（必須）

### ユーザーストーリー 1 — 文章に合うネコチャン絵文字のファイル名を提案してもらう（優先度: P1）

ユーザーが文章をコマンドラインに入力すると、ツールが sphinx-nekochan の
画像ファイル群の中からその文章に最も合う絵文字のファイル名を複数返す。
ユーザーはそのファイル名を Sphinx ドキュメントや他のツールで使用する。

**この優先度の理由**: このツールの存在価値そのものであり、これ単体で価値ある
MVP となる。他のすべての機能はこの中核機能の上に成り立つ。

**独立テスト**: `nekochan-suggest "今日もいい天気ですね"` を実行し、
`hare-nya` や `niko-nya` 等の画像ファイル名の候補リストが標準出力に
表示されれば完結する。

**受け入れシナリオ**:

1. **Given** ポジティブな文章（例：「今日のランチ最高だった！」）、
   **When** コマンドに渡す、
   **Then** 候補が 3 件以上、各候補に画像ファイル名（例: `yatta-nya`）と
   コサイン類似度スコア（0〜1 の浮動小数点）が付いて返る。

2. **Given** 疲れた・落ち込んだトーンの文章（例：「あーもう疲れた…」）、
   **When** コマンドに渡す、
   **Then** トーンに合ったファイル名（例: `kyukei-nya`, `nemui-nya`）が
   候補に含まれる。

3. **Given** 標準入力から文章を渡す（例：`echo "眠い" | nekochan-suggest`）、
   **When** 実行する、
   **Then** コマンドライン引数渡しと同様の形式で結果が返る。

---

### ユーザーストーリー 2 — アノテーションデータベースを構築・更新する（優先度: P2）

管理者（= ツールのセットアップ担当者）が `nekochan-suggest build-annotations`
コマンドを実行すると、sphinx-nekochan の全絵文字（378件）に対してローカル LLM が
アノテーションテキストを生成し、ローカルのデータファイルに保存する。
このデータファイルをもとにクエリ時の提案が行われる。

**この優先度の理由**: ストーリー 1 の提案精度を支える前提処理。初回セットアップと
新しい絵文字が追加された際に実行する。

**独立テスト**: `nekochan-suggest build-annotations` を実行後、
アノテーションファイルに 378 件以上のエントリが保存されれば完結する。

**受け入れシナリオ**:

1. **Given** ネットワーク接続があり aliases.json が取得可能な状態、
   **When** `nekochan-suggest build-annotations` を実行する、
   **Then** 全絵文字のアノテーションファイルが生成・保存される。

2. **Given** アノテーションファイルが既に存在する状態、
   **When** 再実行する、
   **Then** 既存エントリをスキップし、未処理のエントリのみ生成・保存して完了する。

3. **Given** `--dry-run` フラグを付けて実行、
   **When** 実行する、
   **Then** ファイルを保存せず、生成されるアノテーションのサンプルをプレビュー表示する。

---

### ユーザーストーリー 3 — Streamlit GUI で絵文字を提案してもらう（優先度: P3）

ブラウザベースの GUI を使いたいユーザーが `nekochan-suggest-ui` を起動すると、
テキスト入力欄に文章を入力して「提案する」ボタンを押すだけで、
ネコチャン絵文字の候補と類似度スコアが画面に表示される。

**この優先度の理由**: CLI が完成していれば GUI は CLI のラッパーとして実装できる。
非エンジニアのユーザーへのアクセシビリティを高めるが、CLI なしには成立しない。

**独立テスト**: `nekochan-suggest-ui` を起動してブラウザで文章を入力し、
候補が画面に表示されれば完結する。

**受け入れシナリオ**:

1. **Given** アノテーションファイルが存在する状態で `nekochan-suggest-ui` を起動、
   **When** テキスト入力欄に文章を入力して「提案する」ボタンを押す、
   **Then** 画像ファイル名・類似度スコア・ネコチャン画像（GitHub Raw URL から取得）の候補リストが画面に表示される。

2. **Given** アノテーションファイルが存在しない状態、
   **When** ブラウザで提案ボタンを押す、
   **Then** `build-annotations` の実行を促すエラーメッセージが画面に表示される。

---

### ユーザーストーリー 4 — 結果を JSON 形式で受け取る（優先度: P3）

スクリプトや他のツールと組み合わせたいユーザーが `--json` フラグを指定すると、
提案結果が機械可読な JSON 形式で返ってくる。

**この優先度の理由**: 憲法（原則 III）により `--json` は必須。ただし
ユーザー体験の中核はストーリー 1 であるため P3 とする。

**独立テスト**: `nekochan-suggest --json "おはよう"` の出力を JSON パーサーで
処理できれば完結する。

**受け入れシナリオ**:

1. **Given** `--json` フラグと文章を指定、
   **When** 実行する、
   **Then** `{"suggestions": [{"name": "...", "score": 0.87}]}` 形式の
   JSON が標準出力に出力される。

2. **Given** `--json` なしで実行、
   **When** 実行する、
   **Then** 番号付きリスト形式（`N. <ファイル名>  <スコア（小数点以下2桁）>`、例: `1. yatta-nya  0.87`）でテキスト出力される。

---

### ユーザーストーリー 5 — 提案数を指定する（優先度: P3）

ユーザーが `--count` オプションで受け取りたい候補数を指定できる。
デフォルトは 3 件。

**この優先度の理由**: 便利な追加機能だが、固定件数でもツールとして十分機能する。

**独立テスト**: `nekochan-suggest --count 5 "こんにちは"` で 5 件の候補が
返れば完結する。

**受け入れシナリオ**:

1. **Given** `--count 1` と文章を指定、**When** 実行する、
   **Then** 候補が正確に 1 件だけ返る。
2. **Given** `--count` を省略、**When** 実行する、
   **Then** デフォルトの 3 件が返る。

---

### エッジケース

- 空文字列を渡した場合、エラーメッセージを stderr に出力して終了コード 1 で終了する。
- 非常に長い文章（1000 文字超）を渡した場合、適切に処理されるか、
  文字数制限のエラーを返す。
- `--count 0` や負数を渡した場合、バリデーションエラーを返す。
- ローカル LLM サーバーが起動していない場合、接続エラーメッセージと
  起動方法のヒントを stderr に表示する。
- LLM の応答がタイムアウトした場合、エラーメッセージを stderr に出力して
  終了コード 1 で終了する。
- アノテーションファイルが存在しない状態でクエリを実行した場合、
  `build-annotations` の実行を促すメッセージを stderr に表示する。
- `build-annotations` 実行中に一部の絵文字で LLM エラーが発生した場合、
  エラー絵文字をスキップして残りを処理する。完了後にスキップ一覧を stderr に表示する。
- `build-annotations` 実行時に aliases.json のフェッチが失敗した場合（ネットワーク障害・
  非 200 ステータス）、エラーメッセージ（URL・ステータスコード）を stderr に出力して
  終了コード 1 で終了する。
- 日本語以外の文章（英語等）を渡した場合も適切に動作する。

## 要件（必須）

### 機能要件

- **FR-001**: ツールはテキスト入力（コマンドライン引数または標準入力）を受け取り、
  sphinx-nekochan 絵文字の画像ファイル名の候補とコサイン類似度スコアを返さなければならない。
- **FR-002**: 各候補は、画像ファイル名（例: `yatta-nya`）とコサイン類似度スコア
  （0〜1 の浮動小数点）を含まなければならない。
- **FR-003**: デフォルトで 3 件の候補を返さなければならない。
- **FR-004**: `--count N` オプションで候補数を 1〜10 の範囲で指定できなければならない。
- **FR-005**: `--json` フラグを指定すると、結果を JSON 形式で標準出力に
  出力しなければならない（`{"suggestions": [{"name": "...", "score": 0.87}]}`）。
  `--json` なしのテキスト出力は `N. <ファイル名>  <スコア（小数点以下2桁）>` 形式とする
  （例: `1. yatta-nya  0.87`）。
- **FR-006**: エラー（空入力、LLM 接続失敗、アノテーション未生成等）は stderr に
  出力し、終了コード 1 で終了しなければならない。
- **FR-007**: 提案の生成にはローカルで動作する LLM を使用しなければならない。
  デフォルトLLMモデルは `qwen2.5`、埋め込みモデルは `nomic-embed-text`。
  使用するモデルと接続先は TOML 形式の設定ファイル
  （`~/.config/nekochan-suggest/config.toml`）または環境変数で変更できなければならない。
  設定ファイルの読み込みには Python 標準ライブラリの `tomllib` を使用する。
- **FR-008**: 標準入力からのパイプ入力（`echo "文章" | nekochan-suggest`）を
  サポートしなければならない。
- **FR-009**: LLM 応答のタイムアウト時間を設定できなければならない。デフォルトは
  30 秒とし、`--timeout N`（秒単位）オプションまたは環境変数 `NEKOCHAN_TIMEOUT`
  で上書き可能とする。タイムアウト発生時はエラーメッセージを stderr に出力し、
  終了コード 1 で終了する。
- **FR-010**: `build-annotations` サブコマンドにより、sphinx-nekochan の全絵文字に
  対するアノテーションテキストと埋め込みベクトルをローカル LLM で生成し、JSON ファイル
  （`~/.local/share/nekochan-suggest/annotations.json`）に保存しなければならない。
  アノテーションの入力ソースは aliases.json とする。読み書きには Python 標準
  ライブラリの `json` モジュールを使用する。
  aliases.json のフェッチに失敗した場合（ネットワークエラー・非 200 ステータス等）、
  エラーメッセージ（URL・ステータスコード）を stderr に出力し、終了コード 1 で終了する。
  アノテーションは英語・2〜3文で生成する（感情・状況・用途を記述）。
  例: `yatta-nya` → `"A cat celebrating with joy. Use when expressing success, achievement, or excitement. Conveys a positive and energetic mood."`
  処理中は `[N/378] 絵文字名` 形式の進行表示を stderr に `\r` でインライン更新する。追加ライブラリは不要。
  1 件の LLM エラーはスキップして次の絵文字処理を続行する。各絵文字の処理結果はその都度 JSON ファイルに書き込む。
  完了時にスキップした絵文字名をまとめて stderr に出力する。再実行時は既存エントリをスキップする。
  `--dry-run` フラグを指定した場合、先頭 3 件のアノテーションをサンプルとして標準出力にプレビュー表示し、ファイルへの保存は行わない。
- **FR-011**: アノテーションファイルが存在しない場合、クエリ実行時に
  `build-annotations` の実行を促すエラーメッセージを表示しなければならない。
- **FR-012**: クエリ時のマッチングは埋め込みベクトル（embedding）による類似度検索を使用する。
  `build-annotations` 実行時に各アノテーションの埋め込みベクトルを事前生成・保存する。
  クエリ時は入力文章を同じモデルでベクトル化し、コサイン類似度で上位 N 件を返す。
  埋め込みの生成には Ollama Embeddings API（`/api/embeddings`）を使用する。
  デフォルト埋め込みモデルは `nomic-embed-text`（`ollama pull nomic-embed-text` で取得）。
  Ollama への通信には Python 標準ライブラリの `urllib` を使用し、追加 PyPI 依存は不要とする。
  埋め込みモデルは設定ファイルで変更可能とする。
- **FR-013**: `nekochan-suggest-ui` コマンドで Streamlit ベースの GUI を起動できなければならない。
  GUI はテキスト入力欄と「提案する」ボタンを提供し、候補ファイル名・類似度スコア・ネコチャン画像を画面に表示する。
  画像は `https://raw.githubusercontent.com/takanory/sphinx-nekochan/main/sphinx_nekochan/images/<name>.png`
  の URL パターンで構築し `st.image()` で表示する（インターネット接続が必要）。
  内部実装は CLI のライブラリ関数を前提とし、Streamlit は UI 層のみとする。
  アノテーションファイルが存在しない場合はエラーメッセージを画面に表示する。

- **NekochanEmoji**: sphinx-nekochan の絵文字 1 件。
  - `name`: 画像ファイル名（`-nya` 付き、例: `yatta-nya`）
  - `aliases`: エイリアスのリスト（例: `["yatta"]`）
  - `annotation`: LLM が生成した英語の説明テキスト（感情・状況・用途を 2〜3 文で記述）
  - `embedding`: annotation から生成した埋め込みベクトル（浮動小数点の配列）
- **テキスト入力**: ユーザーが提案を求める文章。1 文字以上、1000 文字以下。
- **提案結果**: マッチした NekochanEmoji のファイル名とコサイン類似度スコアのリスト。

### 絵文字データソース

- 絵文字の名前・エイリアス定義:
  `https://github.com/takanory/sphinx-nekochan/blob/main/sphinx_nekochan/data/aliases.json`
- 絵文字の総数: 378 件（2025-11-05 時点）
- 参照ページ: `https://sphinx-nekochan.readthedocs.io/nekochan_emojis.html`

## 成功基準（必須）

### 測定可能な成果

- **SC-001**: LLM の応答生成を除いたツール自体の処理（入力解析・出力整形）が
  1 秒以内に完了する。
- **SC-002**: 入力文章の意味・トーンに合ったファイル名が、提案候補の中に
  80% 以上の確率で含まれる（5 件の異なるトーンの文章でテストした場合）。
- **SC-003**: `--json` 出力が常に有効な JSON として解析できる。
- **SC-004**: エラー発生時に、ユーザーが原因と対処法を理解できるメッセージが
  表示される。
- **SC-005**: `build-annotations` が 378 件すべての絵文字のアノテーションを
  正常に生成・保存できる。

## 前提と想定

- ユーザーはローカルに LLM ランタイム（Ollama）を別途インストール・起動する。
  本ツールは LLM サーバーのセットアップを行わない。
- デフォルトモデルは `qwen2.5`（アノテーション生成用）、`nomic-embed-text`（埋め込み生成用）。
  `ollama pull qwen2.5 && ollama pull nomic-embed-text` で取得できる。
  環境変数 `NEKOCHAN_MODEL` / `NEKOCHAN_EMBED_MODEL` または設定ファイルで別モデルに変更可能。
- Ollama への HTTP 通信には Python 標準ライブラリの `urllib` を使用する。追加 PyPI 依存は不要。
- 「ネコチャン絵文字」とは sphinx-nekochan プロジェクトの画像ファイル群を指す。
  Unicode の猫絵文字やカオモジは対象外。
- アノテーションの生成（`build-annotations`）はインターネット接続が必要。
  クエリ実行（`nekochan-suggest`）はローカルファイルのみで完結する。
- アノテーションファイルの保存先: `~/.local/share/nekochan-suggest/annotations.json`
  （JSON 形式、Python 標準ライブラリの `json` モジュールで読み書き）。
  ファイルには annotation テキストと embedding ベクトルの両方を保存する。
- アノテーションは英語・2〜3文形式で LLM が生成する（感情・状況・使いどころを記述）。
  英語アノテーションにすることで、日本語・英語の両方のクエリに対応できる。
- 対象ユーザーは sphinx-nekochan を使用する日本語を母語とする開発者・ライターを
  想定する。GUI（US4）は非エンジニアのユーザーも想定する。
- GUI 実装には `streamlit` を使用する（`uv add streamlit` でインストール）。
  完全オフライン・ローカル実行のみ。追加依存は標準ライブラリのみである CLI 部分と分離し、
  `streamlit` は GUI エントリーポイントのみで使用する。標準ライブラリ以外の PyPI 依存追加は
  `streamlit` のみとし、既存機能への追加依存は認めない。
