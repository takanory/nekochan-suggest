# 機能仕様書: プロジェクト初期化

**Feature Branch**: `002-project-init`  
**Created**: 2026-03-20  
**Status**: Draft  
**Input**: User description: "プロジェクト初期化を実施"

## ユーザーシナリオとテスト（必須）

### ユーザーストーリー 1 — 開発者がリポジトリをクローンしてすぐに動作確認できる（優先度: P1）

新しい開発者（または CI 環境）がリポジトリをクローンした後、
単一のコマンドで依存関係をインストールし、
`nekochan-suggest --help` が正常に動作する状態に至れる。

**この優先度の理由**: このプロジェクト全体の開発・利用が成立するための大前提であり、
これなしに他のどの機能も開発・テストできない。

**独立テスト**: リポジトリをクローンし、依存関係インストールコマンドを実行後、
`nekochan-suggest --help` が使い方を表示すれば完結する。

**受け入れシナリオ**:

1. **Given** リポジトリをクローンした直後の状態、
   **When** 依存関係インストールコマンドを実行する、
   **Then** エラーなく完了し、`nekochan-suggest` コマンドが使えるようになる。

2. **Given** `nekochan-suggest --help` を実行する、
   **When** コマンドを実行する、
   **Then** ツールの説明・利用可能なオプション（`--count`、`--json`、`--timeout`）が
   標準出力に表示される。

3. **Given** `nekochan-suggest-ui --help` を実行する、
   **When** コマンドを実行する、
   **Then** GUI 起動コマンドの説明が標準出力に表示される。

---

### ユーザーストーリー 2 — プロジェクトのパッケージ構造が整備されている（優先度: P2）

開発者がソースコードを追加・変更できる明確なパッケージ構造が存在し、
CLI の各エントリーポイントが対応するモジュールに接続されている。

**この優先度の理由**: P1 でコマンドが動作した後、実際のビジネスロジックを配置する
パッケージ構造が必要。これが整って初めて他のフィーチャーを実装できる。

**独立テスト**: プロジェクトルートから `nekochan-suggest` と `nekochan-suggest-ui`
の両コマンドがスタブ実装を通じて起動すれば完結する。

**受け入れシナリオ**:

1. **Given** プロジェクトの依存関係がインストール済みの状態、
   **When** `nekochan-suggest "テスト文章"` を実行する、
   **Then** 「未実装（別フィーチャーで実装予定）」などのプレースホルダーメッセージが
   表示される。ビジネスロジックはこのフィーチャーのスコープ外とする。

2. **Given** プロジェクトの依存関係がインストール済みの状態、
   **When** `nekochan-suggest build-annotations` を実行する、
   **Then** サブコマンドとして認識され、プレースホルダーメッセージが表示される。
   アノテーション生成ロジックはこのフィーチャーのスコープ外とする。

---

### ユーザーストーリー 3 — 開発環境でテストが実行できる（優先度: P3）

開発者がコードを変更した後、テストコマンドを実行してパッケージの基本動作を
検証できる。

**この優先度の理由**: 品質保証の基盤。P1・P2 が完成した後に整備する。

**独立テスト**: テスト実行コマンドを使用して、少なくとも 1 つのテストが
パスすれば完結する。

**受け入れシナリオ**:

1. **Given** 依存関係がインストール済みの状態、
   **When** テスト実行コマンドを実行する、
   **Then** テストスイートが実行され、結果（パス/失敗数）が表示される。

---

### ユーザーストーリー 4 — リント・型チェック・フォーマットを 1 コマンドで実行できる（優先度: P3）

開発者がコードを変更した後、ruff（リント・フォーマット）と pyrefly（型チェック）を
単一のコマンドで実行して、コードスタイルと型安全性を確認できる。

**この優先度の理由**: コード品質の一貫性を保つ基盤。後続フィーチャーの実装時に
継続的に使用するため、プロジェクト初期化の段階で整備する。

**独立テスト**: リント・型チェックコマンドを実行し、初期スタブコードに対して
警告・エラーがゼロで完了すれば完結する。

**受け入れシナリオ**:

1. **Given** 依存関係がインストール済みの状態、
   **When** ruff チェックコマンドを実行する、
   **Then** リント違反ゼロで完了する（初期スタブコードに対して）。

2. **Given** 依存関係がインストール済みの状態、
   **When** ruff フォーマットコマンドを実行する、
   **Then** コードが設定済みのスタイルに自動整形される。

3. **Given** 依存関係がインストール済みの状態、
   **When** pyrefly 型チェックコマンドを実行する、
   **Then** 型エラーゼロで完了する（初期スタブコードに対して）。

---

### エッジケース

- Python バージョンが要件外（3.11 未満）の場合、インストール時または実行時に
  わかりやすいエラーメッセージが表示される。
- `nekochan-suggest-ui` のエントリーポイントを使う際、`streamlit` が未インストールの場合は
  Python のデフォルト `ImportError` が表示される。インストール手順は README に記載する。

## 要件（必須）

### 機能要件

- **FR-001**: `nekochan-suggest` コマンドをシステムにインストールできなければならない。
  エントリーポイントは CLI メインモジュールに接続されていること。
- **FR-002**: `nekochan-suggest-ui` コマンドをシステムにインストールできなければならない。
  エントリーポイントは Streamlit GUI モジュールに接続されていること。
- **FR-003**: `nekochan-suggest` は `[テキスト]`（位置引数）と `build-annotations`
  （サブコマンド）の 2 つの動作モードをサポートしなければならない。
- **FR-004**: パッケージは Python 3.11 以上を必要とする旨を明示しなければならない。
- **FR-005**: パッケージのコア機能（CLI・クエリロジック・アノテーション生成）は
  外部ライブラリへの依存なしに動作しなければならない。
  `streamlit` は `pyproject.toml` のオプション依存グループ（例: `gui`）として定義し、
  インストール手順は README に記載する。独自のエラーハンドリングは行わず、
  未インストール時は Python のデフォルト `ImportError` に委ねる。
- **FR-006**: プロジェクトのルートに `README.md` が存在し、インストール手順と
  基本的な使い方（`nekochan-suggest "文章"`）が記載されていなければならない。
- **FR-007**: `nekochan-suggest --help` の実行で、コマンド説明・引数・オプション
  （`--count`、`--json`、`--timeout`）の一覧が表示されなければならない。
- **FR-008**: `ruff` をリンター・フォーマッターとして開発依存に含め、設定を
  `pyproject.toml` の `[tool.ruff]` セクションに記述しなければならない。
  対象ディレクトリ・Python バージョンを設定し、有効ルールセットは
  `["E", "F", "I", "N", "UP"]`（pycodestyle エラー・ Pyflakes・isort・命名規則・Pythonバージョンアップグレード）を
  設定すること。
- **FR-009**: `pyrefly` を型チェッカーとして開発依存に含め、設定を
  `pyproject.toml` の `[tool.pyrefly]` セクション（または専用設定ファイル）に
  記述しなければならない。少なくともチェック対象ディレクトリを設定すること。
- **FR-010**: ruff チェック・ruff フォーマット・pyrefly 型チェック・pytest の
  各コマンドを単一ステップで呼び出せるよう、プロジェクトルートに `Makefile` を配置しなければならない。
  少なくとも `lint`（ruff check）・`format`（ruff format）・`typecheck`（pyrefly）・
  `test`（pytest）・`check`（上記 4 つを順に実行）の各ターゲットを定義すること。
  CI・ローカル共通の同一コマンドで実行できること。

### キーエンティティ

- **パッケージ**: `nekochan_suggest` として配置される Python パッケージ（フラット構成）。
  `__init__.py`・`cli.py`・`query.py`・`annotations.py`・`ui.py` の 5 モジュールで構成する。
  各モジュールの責務: `cli.py` = CLI エントリーポイント・引数解析、
  `query.py` = 埋め込みベクトル検索ロジック、
  `annotations.py` = アノテーション生成・保存ロジック、
  `ui.py` = Streamlit GUI。
- **エントリーポイント**: `nekochan-suggest`（`cli.py` 経由）と `nekochan-suggest-ui`（`ui.py` 経由）
  の 2 つのコマンドラインコマンド。

## 成功基準（必須）

### 測定可能な成果

- **SC-001**: リポジトリクローン後、1 コマンドだけで開発・利用環境が整う
  （追加の手動設定が不要）。
- **SC-002**: `nekochan-suggest --help` が体感即時（1 秒以内）で応答を返す。
- **SC-003**: パッケージのインポートがエラーなく完了し、
  CLI の全サブコマンドが認識される。
- **SC-004**: `streamlit` 未インストールの環境でも `nekochan-suggest`
  コマンドが正常に起動する（GUI 依存を分離できている）。
- **SC-005**: 初期スタブコードに対して ruff チェックと pyrefly 型チェックが
  警告・エラーゼロで完了する。

## 前提と想定

- プロジェクトのパッケージ管理ツールとして `uv` を使用する。
  インストール・実行コマンドは `uv sync` / `uv run` を基本とする。
- Python バージョン要件: 3.11 以上（`tomllib` が標準ライブラリに含まれるバージョン）。
- パッケージ名: `nekochan-suggest`（ハイフン区切り、PyPI 配布は想定しない）。
- コア機能の外部 PyPI 依存は 0 とする（標準ライブラリのみ）。
  唯一の例外は GUI 用の `streamlit`（オプション依存）。
- テストフレームワークは `pytest` を開発依存として使用する。
- リンター・フォーマッターとして `ruff` を開発依存として使用する。
- 型チェッカーとして `pyrefly` を開発依存として使用する。
- 各開発タスク（lint・format・typecheck・test）の実行手段として `Makefile` を使用する。
  `make` は GitHub Actions の ubuntu-latest / macos-latest ランナーに標準搭載されており、追加インストール不要。
