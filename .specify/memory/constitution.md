# nekochan-suggest 憲法

## コア原則

### I. Python ファースト・シンプリシティ

実装言語は Python 3.13+ のみとする。
標準ライブラリを最優先で使用すること。サードパーティ依存関係は計画書において
明確に正当化されなければならない（「あると便利」という理由は認めない）。
YAGNI 原則を適用する：機能は具体的なユーザーストーリーが要求した時にのみ追加する。
各モジュールは単一の明確な責務を持つこと。
設定の読み込み以外でのグローバルな可変状態は禁止とする。

**理由**: 小さく焦点を絞ったツールは、一人の開発者が長期にわたって保守しやすい。
不要な依存関係は保守負債である。

### II. テストファースト（絶対遵守）

非自明なロジックにはすべて TDD を適用すること：

1. テストを書く。
2. テストが失敗することを確認する（レッド）。
3. テストをパスする最小限のコードを実装する（グリーン）。
4. リファクタリングする。

テストランナーは `pytest` とする。コアモジュールは行カバレッジ ≥ 80% を維持すること。
LLM 呼び出しを含むコードは `unittest.mock` でモックし、CI にモデルの起動は不要とすること。
テストはリポジトリにコミットし、PR マージ前にすべてパスしなければならない。

**理由**: テストなしでは将来の LLM 統合方式の変更が危険になる。
モックによって CI を高速かつ安定した状態に保てる。

### III. CLI ファースト・インターフェース

主要なユーザーインターフェースはコマンドラインツールとする。
プロトコル：`stdin` / `--args` → `stdout`、エラーと診断情報 → `stderr`。
構造化出力を持つすべてのコマンドに `--json` フラグを実装しなければならない。
ライブラリ API は CLI を反映すること。すべての CLI コマンドは直接インポートして
呼び出し可能な純粋な Python 関数をラップする形とする。

**理由**: CLI ツールはスクリプトや他のツールとの組み合わせが容易である。
ライブラリ API を並行提供することで、サブプロセスのオーバーヘッドなしに再利用できる。

### IV. 可観測性と型安全性

すべての公開関数とクラスメソッドに型ヒント（PEP 484）を付与しなければならない。
`pyrefly` を厳格モードで使用し、`src/` ツリー全体をチェックすること。
構造化ロギングは Python 標準の `logging` モジュールを使用し、ログレベルは設定可能と
すること。ライブラリコード内での `print()` 使用は禁止とする。
エラーはデバッガーなしに診断できる十分なコンテキストとともに表面化させること。
サイレントな失敗（裸の `except: pass` や例外の握り潰し）は禁止とする。

**理由**: 型安全性と構造化ログは最も安価なドキュメントの形式であり、
LLM 関連のエッジケースをデバッグする際の最初の防衛線となる。

### V. 日本語ドキュメント

すべてのドキュメント（仕様書、計画書、コードコメント、README、
コミットメッセージを除く自然言語の説明）は日本語で記述しなければならない。
コードコメント（インラインコメント、docstring）も日本語で記述すること。
コミットメッセージのプレフィックスは Conventional Commits の英語形式
（`feat:`, `fix:` 等）を維持し、説明文は日本語とする。
API 公開インターフェースの docstring は日本語を必須とする。

**理由**: このプロジェクトの主要な開発者・利用者は日本語話者である。
日本語でのドキュメントは理解の精度を高め、認知負荷を下げる。
英語と日本語の混在は一貫性を損なうため、日本語に統一する。

## テクノロジースタック

- **言語**: Python 3.13+
- **LLM ランタイム**: ローカルのみ（Ollama 推奨；直接呼び出し可）
- **CLI フレームワーク**: `argparse`（標準ライブラリ）または `typer` ─ 機能計画で要正当化
- **テスト**: `pytest`, `pytest-cov`, `unittest.mock`
- **型チェック**: `pyrefly`（厳格モード）
- **リント／フォーマット**: `ruff`（lint + format、flake8/black/isort の代替）
- **依存関係管理**: `uv` + `pyproject.toml`（PEP 621）

これらのデフォルトからの逸脱は、該当機能の計画書に明確な正当化とともに記載すること。

## 開発ワークフロー

1. **仕様ファースト**: 実装開始前に `spec.md` が存在しなければならない。
2. **計画ゲート**: Constitution Check に合格した `plan.md` が承認されてから、
   プロダクションコードを書き始める。
3. **TDD サイクル**: 原則 II を厳格に遵守する。
4. **PR チェックリスト**:
   - すべてのテストがパス（`pytest`）。
   - 型チェックがパス（`pyrefly src/`）。
   - リンターがパス（`ruff check . && ruff format --check .`）。
   - Constitution Check（原則 I〜V）を再確認。
   - 新規ドキュメント・コメントが日本語で記述されている。
5. **メインブランチへの force-push は禁止**。
6. **コミットメッセージ**: Conventional Commits 形式
   （`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`）
   プレフィックスの後の説明文は日本語とする。

## ガバナンス

この憲法はリポジトリ内の他のすべての慣習、ガイドライン、選好に優先する。
改定時は以下のセマンティックバージョニングに従って `CONSTITUTION_VERSION` を増やすこと：

- **MAJOR**: コア原則の削除または再定義。
- **MINOR**: 新原則・セクションの追加、または実質的な説明の拡充。
- **PATCH**: 明確化・表現の修正・非意味論的な改善。

ソースコードに触れるすべての PR は、原則 I〜V への準拠を確認する
「Constitution Check」セクションを含まなければならない。
いずれかの原則に違反する複雑さは、PR オープン前に計画書に書面で正当化すること。
憲法は各主要機能サイクルの開始時に見直し、プロジェクトのスコープや制約が
大幅に変化した場合は改定する。
